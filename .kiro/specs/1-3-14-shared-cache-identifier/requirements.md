# Requirements Document

## Introduction

This feature adds a `sharedCacheId` parameter to `Cache.init()` that allows multiple Lambda functions to share cached data by overriding the default per-function salt used in cache key hash generation. Currently, `Cache.generateIdHash()` uses `process.env.AWS_LAMBDA_FUNCTION_NAME` as the salt, meaning two Lambda functions fetching the same data from the same endpoint produce different cache key hashes and store separate entries in DynamoDB/S3. The `sharedCacheId` parameter provides a clean, opt-in mechanism for functions to share cache entries without overriding AWS-managed environment variables.

## Glossary

- **Cache_Module**: The `cache` export of the @63klabs/cache-data NPM package, providing distributed caching using DynamoDB and S3.
- **Cache_Class**: The public static class in the Cache_Module that provides the high-level caching API including `init()`, `generateIdHash()`, and `info()`.
- **Cache_Hash**: The deterministic hash generated by `Cache.generateIdHash()` from a connection object and a salt value, used as the cache key in DynamoDB/S3.
- **Salt**: A string value mixed into the hash computation to differentiate cache entries. Currently defaults to `process.env.AWS_LAMBDA_FUNCTION_NAME`.
- **Shared_Cache_Id**: An optional string parameter provided to `Cache.init()` that overrides the default Lambda function name salt, enabling multiple functions to produce identical cache hashes for the same data.
- **CACHE_SHARED_ID**: An environment variable that serves as a fallback source for the shared cache identifier when the `sharedCacheId` parameter is not provided to `Cache.init()`.
- **Lambda_Function_Name**: The value of `process.env.AWS_LAMBDA_FUNCTION_NAME`, an AWS-managed environment variable identifying the current Lambda function.
- **Init_Parameters**: The configuration object passed to `Cache.init()` containing settings such as `secureDataKey`, `idHashAlgorithm`, and the new `sharedCacheId`.

## Requirements

### Requirement 1: Shared Cache Identifier Parameter

**User Story:** As a developer deploying multiple Lambda functions that access the same cached data, I want to provide a `sharedCacheId` to `Cache.init()`, so that all functions sharing the same identifier produce identical cache hashes for the same request.

#### Acceptance Criteria

1. WHEN `sharedCacheId` is provided as a string in Init_Parameters, THE Cache_Class SHALL use that value as the Salt in `generateIdHash()` instead of the Lambda_Function_Name.
2. WHEN `sharedCacheId` is provided as an empty string in Init_Parameters, THE Cache_Class SHALL use an empty string as the Salt (no salt applied).
3. WHEN `sharedCacheId` is not provided in Init_Parameters and the CACHE_SHARED_ID environment variable is set, THE Cache_Class SHALL use the CACHE_SHARED_ID value as the Salt.
4. WHEN `sharedCacheId` is not provided in Init_Parameters and the CACHE_SHARED_ID environment variable is not set, THE Cache_Class SHALL use the Lambda_Function_Name as the Salt (current default behavior).

### Requirement 2: Parameter Priority

**User Story:** As a developer, I want the `sharedCacheId` parameter to take priority over the environment variable, so that I can override environment-level configuration at the code level when needed.

#### Acceptance Criteria

1. WHEN both `sharedCacheId` parameter and CACHE_SHARED_ID environment variable are present, THE Cache_Class SHALL use the `sharedCacheId` parameter value as the Salt.
2. WHEN `sharedCacheId` parameter is set to an empty string and CACHE_SHARED_ID environment variable is set to a non-empty value, THE Cache_Class SHALL use the empty string as the Salt (parameter takes priority).
3. THE Cache_Class SHALL resolve the effective Salt using the following priority order: `sharedCacheId` parameter, then CACHE_SHARED_ID environment variable, then Lambda_Function_Name, then empty string.

### Requirement 3: Cache Hash Determinism with Shared Identifier

**User Story:** As a developer, I want two Lambda functions configured with the same `sharedCacheId` and making the same request to produce identical cache hashes, so that they read and write to the same cache entry.

#### Acceptance Criteria

1. FOR ALL valid connection objects, two Cache_Class instances initialized with the same `sharedCacheId` value SHALL produce identical Cache_Hash values from `generateIdHash()`.
2. FOR ALL valid connection objects, two Cache_Class instances initialized with different `sharedCacheId` values SHALL produce different Cache_Hash values from `generateIdHash()`.
3. FOR ALL valid connection objects, a Cache_Class instance initialized with a `sharedCacheId` value SHALL produce a different Cache_Hash than one using the default Lambda_Function_Name salt (unless the `sharedCacheId` equals the Lambda_Function_Name).

### Requirement 4: Cache Info Exposure

**User Story:** As a developer, I want `Cache.info()` to include the effective shared cache identifier, so that I can verify the configuration and debug cache sharing issues.

#### Acceptance Criteria

1. THE Cache_Class `info()` method SHALL include a `sharedCacheId` property in the returned object.
2. WHEN a Shared_Cache_Id is configured (via parameter or environment variable), THE `info()` method SHALL return the effective Shared_Cache_Id value in the `sharedCacheId` property.
3. WHEN no Shared_Cache_Id is configured, THE `info()` method SHALL return `null` in the `sharedCacheId` property.

### Requirement 5: Backwards Compatibility

**User Story:** As an existing user of the package, I want my current code to work without modification after this update, so that I can upgrade without breaking changes.

#### Acceptance Criteria

1. WHEN `sharedCacheId` is not provided in Init_Parameters and CACHE_SHARED_ID is not set, THE Cache_Class SHALL produce identical Cache_Hash values to the current implementation for all inputs.
2. THE Cache.init() method signature SHALL remain backwards compatible — existing Init_Parameters objects without `sharedCacheId` SHALL continue to work without modification.
3. THE Cache.generateIdHash() method signature SHALL remain unchanged.
4. THE `sharedCacheId` parameter SHALL NOT affect any other Cache_Module behavior (encryption, expiration, storage routing, in-memory caching).

### Requirement 6: Input Validation

**User Story:** As a developer, I want clear feedback if I provide an invalid `sharedCacheId` value, so that I can correct configuration errors quickly.

#### Acceptance Criteria

1. WHEN `sharedCacheId` is provided as a non-string, non-null, non-undefined value (e.g., number, object, array), THE Cache_Class SHALL throw a descriptive Error during `init()`.
2. WHEN `sharedCacheId` is provided as `null` or `undefined`, THE Cache_Class SHALL treat it as not provided and fall through to the environment variable or default behavior.
3. THE Cache_Class SHALL accept any string value for `sharedCacheId` including empty string, whitespace, and strings with special characters.

### Requirement 7: User Documentation Update

**User Story:** As a developer adopting the shared cache feature, I want clear documentation in `docs/features/cache/README.md` explaining how to configure and use `sharedCacheId`, so that I can set up cross-function cache sharing without reading source code.

#### Acceptance Criteria

1. THE `docs/features/cache/README.md` file SHALL include a section documenting the `sharedCacheId` option in the Configuration Options table with its type, default, and description.
2. THE documentation SHALL include a usage example showing how to configure `sharedCacheId` in `Cache.init()` for cross-function cache sharing.
3. THE documentation SHALL explain the priority resolution order: `sharedCacheId` parameter → `CACHE_SHARED_ID` environment variable → `AWS_LAMBDA_FUNCTION_NAME` → empty string.
4. THE documentation SHALL include the `CACHE_SHARED_ID` environment variable in the environment variables section.
5. THE documentation SHALL include a warning that sharing cache means sharing expiration — if Function A writes a cache entry that expires in 10 minutes, Function B will see that same expiration.

### Requirement 8: TypeScript Type Definition Update

**User Story:** As a TypeScript consumer, I want the `sharedCacheId` option reflected in the type definitions, so that I get IntelliSense support for the new parameter.

#### Acceptance Criteria

1. THE Declaration_File SHALL include `sharedCacheId?: string` in the `CacheInitParameters` interface (or equivalent init parameter type).
2. WHEN a Consumer constructs Init_Parameters, THE IntelliSense SHALL display `sharedCacheId` as an available optional property with a description.
3. THE `info()` return type SHALL include `sharedCacheId: string | null`.
